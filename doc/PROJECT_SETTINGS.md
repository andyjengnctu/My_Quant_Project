# 專案設定

## 執行紀律
- 永遠以本輪使用者最新提供的程式、ZIP、檔案與 `/doc/PROJECT_SETTINGS.md` 為唯一基準。
- 除非條文明示例外，本檔所有條文均為強制規則；assistant 不得忽略、弱化、選擇性遵守、以慣例覆蓋、或自行推定例外。
- 每次開始前，必須先回報：ZIP 檔名、SHA256、全新解壓目錄、已讀 `/doc/PROJECT_SETTINGS.md`。
- 每次開始前，必須先盤點必要資產：`<repo>/models/best_params.json`、optimizer DB、`requirements` 所需套件。測試資料規則與搜尋順序以 D 節為準。
- 不得沿用前一輪解壓目錄、暫存、舊版本結論或舊修補結果。
- 使用者要求「再檢查」、「最嚴格檢查」或「一次修好」時，必須先完成完整問題盤點，再開始修補；不得邊測邊修。
- 若判定本輪仍有問題，必須提出本輪 ZIP 的實際檔案證據與行號，不得直接歸因於使用者。
- 不得把 assistant 自己的解壓、暫存、容器、套件、資料遺失或測試操作問題歸因於使用者。
- 只提供客觀分析與建議，不奉承；先檢討自己，再提出質疑。
- 回答必須明確、精簡、避免重複。
- AI 註解與使用者註解必須明確區分；只能刪 AI 註解，不可刪使用者註解，不可擅改或刪除原本正確的程式碼。
- 提供程式碼片段時，必須先給可直接搜尋定位的完整舊片段，再給可直接貼上的新片段。
- 提供修補 ZIP 時，只能包含有修改的檔案，並列出：修改檔案、修改原因、重現方式、回歸結果。
- 如架構調整需刪檔，須提供可執行的command，避免使用者手動刪錯。
- 修改 `/doc/PROJECT_SETTINGS.md` 時，必須以GPT可明確遵守、使用者易讀易維護、兼顧後續泛用性為原則，應保持文字精簡、嚴禁重複條文，除非確有必要不新增分類。
- 當 assistant 執行超過10分鐘時，分析其原因，提供改善方法。

## 驗證流程

- 每次修補前，必須先完成完整問題盤點；至少涵蓋：
   - 靜態掃描、`py_compile`、`compileall`
   - `--help`
   - CLI 缺值 / 空值 / 非法值
   - 缺參數檔 / 壞參數檔
   - 壞 DB / 空 DB
   - export-only 路徑
   - headless / stdout-stderr 輸出順序
   - reduced 真實資料主流程
   - reduced consistency
   - 已修問題回歸
   - 錯誤路徑驗證
- 驗證順序必須是：先靜態，靜態全過後才做動態。
- 最嚴格檢查必須一次找出本輪所有明確問題，包含程式、工具、統計口徑、交易限制、初始化順序、錯誤路徑與 headless 輸出問題。
- 修補後必須整輪回歸，不得只驗單一路徑。
- 當使用者提供最新程式 ZIP 與對應的本地 `apps/test_suite.py` bundle，且 bundle 可證明與 ZIP 一致時，assistant 不得重複執行 bundle 已覆蓋且為 PASS 的驗證；僅可補驗 bundle 缺項、FAIL、與 ZIP／報告／程式內容不一致、本輪修改點未覆蓋、或使用者明確要求的項目。補驗前必須先完成必要環境檢查；assistant 自身環境或操作問題不得視為程式缺陷，亦不得作為重跑 bundle 已覆蓋且為 PASS 項目的理由。若 bundle 已明確列出本輪修改點對應驗證且為 PASS，不得再做函式級、子程序級或局部抽查；除非有不一致證據或使用者明確要求。
- 只有在「已完成完整盤點、問題已一次列全、已全部修補、已整輪回歸、且無未揭露未執行驗證項或環境問題」時，才可寫「本輪已全部修好」。

## 測試資料與環境規則

- `tw_stock_data_vip_reduced` 為獨立測試資料，不包含於程式包 ZIP；測試資料固定來自本投資專案的專案資料 `data.zip`。
- reduced 驗證前，必須先由 `data.zip` 取得 `tw_stock_data_vip_reduced`。
- 本地端運行時，除了`validate_consistency.py`，其它在apps中的程式都用full dataset; 
- 驗證時節省時間驗證，一律以 `tw_stock_data_vip_reduced` 為準，不做 `tw_stock_data_vip` full dataset 測試。
- 只有在已確認 `data.zip` 中不存在或不可用 `tw_stock_data_vip_reduced` 時，才可寫「reduced 動態驗證未執行」；不得解讀為程式包缺檔、專案缺陷或使用者未提供資料。
- 若需外部套件，應依 `requirements` 自行安裝後再驗證；不得因缺套件就略過本應執行的動態測試。

## Coding 原則

- 單一真理來源：相同邏輯不得重複實作。
- 統計口徑必須完全一致：成交、未成交、miss buy、EV、勝率、Round-Trip PnL 不得分叉。
- 避免 magic number；公式與參數必須可解釋。
- 禁止裸 except；所有異常必須可追蹤。
- 架構調整不得明顯犧牲效率；若提高未來策略修改或 ML / DRL / LLM 升級複雜度，必須先明確說明。
- 正式入口集中於 `apps/`；`core/` 只放核心規則與共用計算；`tools/` 只放驗證、除錯與開發輔助工具。
- 拆分、合併、移動、重新命名檔案時，必須遵守：單一職責、上層呼叫下層、禁止反向依賴、禁止循環依賴、禁止規則分叉、禁止重複實作。
- 架構或模組責任有變動時，必須同步更新 `doc/ARCHITECTURE.md` 與 `doc/CMD.md`。

## 交易與策略原則
 
- 杜絕未來函數：不可偷看未來資料。
- 同 K 棒同時觸發停損 / 停利時，一律以最壞停損情況計算。
- 資金與權益曲線一律以扣除手續費、稅金後的淨值為準。
- 半倉停利僅視為現金回收，尾倉結算才算完整 Round-Trip PnL。
- 只能盤前掛單；盤中不得新增、改單、換股。
- 不得當沖；買入當日不可賣出；當日賣出回收資金不得於當日再投入。
- 若某檔當日未成交，不得於同一交易日盤中自動改掛其他股票。
- 停利 / 停損只能對已持有部位預先設定。
- 候選資格、是否掛單、是否成交、是否 miss buy、歷史績效統計，必須分層定義，不得混用。
- 單股回測不得使用該檔自身歷史績效 filter 作為買入閘門；歷史績效 filter 只能用於投組層。

## 輸出與打包原則

- `outputs/` 根目錄只放工具分類資料夾；各工具輸出必須落到各自資料夾，禁止再把檔案散落到 `outputs/` 根目錄。
- `apps/test_suite.py` 不論 PASS / FAIL，都必須先在 staging 整理結果再打成單一 bundle；歷史 bundle 保留於 `outputs/local_regression/`、專案根目錄只保留最新一份同名 copy，且預設只保留 minimum set、詳細除錯材料僅在 FAIL 或指定 debug 模式時納入 bundle。
- `apps/test_suite.py` 結束後必須自動執行 output retention，至少同時採用「最近 N 份」與「最多 D 天」雙門檻，且規則集中於單一來源；變更任何輸出路徑、bundle 內容或保留規則時，必須同步更新 `doc/CMD.md`、`doc/ARCHITECTURE.md` 與本檔。
