# 專案設定
1. 永遠以使用者提供的最新程式 / 最新 ZIP 或上傳檔案為準修改，並總是讀取 `/doc/PROJECT_SETTINGS.md` 中最新的專案設定，依最新設定執行。

## A. 回覆原則
1. AI 註解與使用者註解必須明確區分；只能刪 AI 註解，不可刪使用者註解，嚴禁擅改或刪除原本正確的程式碼。
2. 只提供客觀分析與建議，不奉承。
3. 但需先檢討自己，才可提出合理質疑。
4. 何時候都要保持禮貌，語氣要客氣。
5. 重點回答，避免重複說明相同概念。
6. 回答要明確，是就是是、非就是非，避免似乎、幾乎、比較好、比較差這種模稜兩可的用語。
7. 當使用者要求「最嚴格檢查」時，須一次找出所有問題，包含程式、工具、統計口徑、交易限制符合性，包含全檔靜態審核、py_compile 語法檢查、合成案例等驗證，如靜態過了才做動態檢查。
8. 需做動態測試，測試資料預設放在 `/data/tw_stock_data_vip_reduced`；若執行環境不存在 `/data`，則退回專案根目錄下的 `data/tw_stock_data_vip_reduced`。程式包中不會放完整測試資料；如需外部套件（如 `optuna`）就自行安裝，環境設定寫在 `requirements` 資料夾中。
9. 檢查有問題就直接提供修改，須徹底修好，避免產生新問題。
10. 如果是提供更新的程式碼片段，總是先提供明確要被取代的完整片段（需確認過可直接貼上搜尋就找到），縮排也必須可直接貼上使用，讓使用者可以無腦地 copy-search-paste 就完成。
11. 如果是提供更新的程式碼 ZIP 檔，只給有改的檔案就好，放在正確的資料夾目錄中，並列出改了哪些檔，讓使用者可以直接 copy-paste 取代舊檔案。

## B. 版本紀律（強制，避免AI幻覺）
1. 每次檢查或修改，只能以本輪使用者最新上傳的 ZIP 為唯一基底。
2. 每次開始前，必須解壓到全新工作目錄，不得沿用舊目錄或舊暫存。
3. 每次開始前，必須先回報：ZIP 檔名、SHA256、解壓目錄，並確認已讀 `/doc/PROJECT_SETTINGS.md`。
4. 若判定「未整合修正」或「退回舊版」，必須先提出當前 ZIP 的實際檔案證據與行號，不得直接歸因於使用者。
5. 每次修補後，必須列出修改檔案、修改原因、重現方式、回歸驗證結果。
6. 每次修補必須做整輪回歸：靜態檢查、語法檢查、consistency、主流程、已修問題回歸、錯誤路徑與 headless 驗證。
7. 使用者要求「再檢查」或「一次修好」時，必須完整重檢並一次修完本輪所有明確問題，不得只修單點。
8. 不得把 assistant 自己的版本對齊、解壓、暫存、環境問題歸因於使用者。

## C. Coding 原則
1. 單一真理來源：相同邏輯不得在不同地方重複實作。
2. 統計口徑絕對一致：回測、優化、模擬、驗證工具中的成交、未成交、miss buy、EV、勝率、Round-Trip PnL 定義必須完全一致。
3. 避免 magic number；公式與參數需具數學意義與可解釋性。
4. 禁止裸 except，所有異常必須可追蹤。
5. 架構調整不可明顯犧牲效率；若增加未來策略修改或 ML / DRL / LLM 升級複雜度，必須先明確說明。
6. 檔案拆分與結構原則
    - 何時該拆: 單檔程式應控制複雜度與職責範圍；若單一檔案職責混雜、規則耦合、明顯過大，或已影響維護、驗證、除錯與後續擴充，應依單一職責、單一真理來源與穩定介面進行適度拆分；不得為拆分而碎片化，也不得破壞統計口徑一致性、交易規則一致性、測試可重現性，或明顯犧牲效率。
    - 依什麼判斷拆: 拆分判斷應以降低規則耦合、提升可驗證性、避免口徑分叉，以及維持未來 ML / DRL / LLM 模組接入、替換、實驗與回歸驗證的相容性為目的；不得僅以行數作為唯一拆分依據。
    - 檔案/資料夾結構原則: 正式執行入口應集中於固定入口層（目前為 `apps/`），讓使用者可一眼辨識可直接執行的主程式。`core/` 僅放核心規則與共用計算；`tools/` 僅放除錯、驗證與開發輔助工具，且應再依職責分子資料夾（如 `tools/validate/`、`tools/debug/`）；helper 模組不得與正式入口混放同層。
    - 命名與對外介面原則: 新增檔案與模組應優先使用職責語意命名，不得使用版本前綴；對外穩定介面應優先透過套件 `__init__.py` 或 façade 檔統一匯出，避免上層直接依賴過深子模組路徑。若未來要處理歷史函式／類別識別字中的版本字樣，須另案評估，不得在一般功能修正中零碎更改。
    - 拆完後架構與文件要求: 檔案拆分、合併、移動、重新命名、建立新入口或調整模組責任後，應採分層、單一職責與單純依賴方向架構：核心規則與計算、流程編排、工具層、參數／設定／驗證、測試與一致性驗證應分離管理，並以上層呼叫下層為原則，禁止反向依賴、循環依賴、重複實作與規則分叉；凡涉及檔案結構或模組責任變動，必須同步更新 `doc/ARCHITECTURE.md` 與 `doc/CMD.md`，確保文件與實際程式一致。
    - 舊檔清理要求: 當架構改變導致舊入口、舊 helper、舊 wrapper、舊檔名或舊路徑已不再是正式結構一部分時，必須明確判定哪些檔案應保留、哪些檔案必須刪除；若決定刪除，應在修改說明、補丁清單或回歸報告中明確列出應刪除的舊檔，並同步移除相關文件、命令、匯入路徑與程式內提示文字中的舊引用，避免新舊路徑並存造成混淆。除非有明確過渡需求與期限，否則不得長期保留僅為相容而存在的舊檔。

## D. 交易原則
1. 杜絕未來函數：不可偷看未來資料。
2. 同 K 棒同時觸發停損 / 停利時，一律以最壞停損情況計算。
3. 資金與權益曲線一律以扣除手續費、稅金後的淨值為準。
4. 半倉停利僅視為現金回收，尾倉結算才算完整 Round-Trip PnL。
5. 只能盤前掛單；盤中不得新增、改單、換股。
6. 買入股數上限、總資金占用、最大持倉數、單筆風險曝險，都必須在盤前即可確定。
7. 不得當沖；買入當日不可賣出。
8. 當日賣出回收資金，不得於當日再投入。
9. 若某檔當日未成交，不得於同一交易日盤中自動改掛其他股票。
10. 停利 / 停損僅能對已持有部位預先設定。

## E. 候選 / 延續候選 / Miss buy 定義
1. 當日買入候選資格定義：某檔股票於某交易日，必須同時符合技術買入訊號與歷史績效門檻，才算當日買入候選。
2. 每日買入候選池由當日新取得候選資格之標的與延續候選共同組成。
3. 是否掛單由當日盤前排序、可用資金、持倉名額、風險股數上限共同決定。
4. 延續候選定義：某檔股票只要曾於先前交易日出現技術買入訊號但尚未買入，後續交易日只要仍符合當日延續技術條件、歷史績效門檻與風險限制，即可重新進入候選池參與排序。
5. 某檔前一日未被選到、資金不足、名額不足、或有效掛單未成交，後續交易日只要仍符合當日延續技術條件、歷史績效門檻與風險限制，皆可再次進入候選池參與排序。
6. 延續候選不是特殊通道；不得因曾進入候選池、曾掛單、曾未成交，就繞過當日排序直接保送買進。
7. miss buy 定義：已於盤前保留資金與名額，且已形成有效限價買單，但盤中未成交。
8. 以下情況都不算 miss buy：排序未選上、資金不足、持倉名額不足、風險股數為 0、先達停損而放棄進場。
9. 候選資格、是否掛單、是否成交、是否 miss buy，必須分開定義，禁止混用。

## F. 策略邏輯
1. 單股回測不使用該檔股票自身的歷史績效 filter 作為買入閘門；單股回測只用來驗證核心策略本身對該檔股票的適用性。
2. 單股回測必須完整依技術訊號、盤前掛單、成交／未成交、停損／停利規則產生交易歷程，不可因該檔自身歷史 EV、勝率或交易次數而提前封鎖後續進場。
3. 歷史績效 filter 僅能用於投組層的候選資格、排序、是否掛單等決策；不得回頭影響單股回測本身的交易生成。
4. 投組層使用的歷史績效統計來源，應來自單股回測產生的完整交易結果；不得使用投組自身是否實際買到作為統計母體。
5. 技術訊號、候選資格、是否掛單、是否成交、miss buy、歷史績效統計，必須分層定義，禁止混用。

## G. 專案特例
1. `apps/portfolio_sim.py` 自動開瀏覽器暫時允許。
2. 暫時都只使用還原價，不考慮 raw。
3. 資料集預設：若執行環境存在 `/data`，則 `apps/validate_consistency.py` 預設使用 `/data/tw_stock_data_vip_reduced`，`apps/portfolio_sim.py`、`apps/vip_scanner.py`、`apps/ml_optimizer.py` 預設使用 `/data/tw_stock_data_vip`；若執行環境不存在 `/data`，則自動退回專案根目錄下的 `data/tw_stock_data_vip_reduced` 與 `data/tw_stock_data_vip`。除 consistency 外不在一般 UI 額外詢問資料集；若需切換，只能透過 CLI `--dataset` 或環境變數覆蓋，且不得把未顯示資料集選單視為問題。
