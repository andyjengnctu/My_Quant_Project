# 專案設定

1. 永遠以本輪使用者最新提供的程式、ZIP、檔案與 `/doc/PROJECT_SETTINGS.md` 為唯一基準執行。
2. 不得沿用前一輪解壓目錄、暫存、舊版本結論或舊修補結果。

## A. 執行紀律（強制）

1. 每次檢查或修改前，必須先回報：
   - ZIP 檔名
   - SHA256
   - 全新解壓目錄
   - 已讀 `/doc/PROJECT_SETTINGS.md`

2. 每次開始前，必須先做資產盤點，並明確區分「外部環境資產」與「專案內資產」：
   - `/data/tw_stock_data_vip`
   - `/data/tw_stock_data_vip_reduced`
   - `<repo>/data/tw_stock_data_vip`
   - `<repo>/data/tw_stock_data_vip_reduced`
   - `<repo>/models/best_params.json`
   - optimizer 使用的 DB
   - `requirements` 中主流程所需套件
   若缺任何必要資產，必須明確列為「資料阻塞」或「環境阻塞」，不得混稱為已通過。

3. 只要使用者要求「再檢查」、「最嚴格檢查」或「一次修好」，就必須先完成完整問題盤點，再開始修補；不得邊測邊修、修一點算一點。

4. 若判定「未整合修正」、「退回舊版」或「本輪仍有問題」，必須提出本輪 ZIP 的實際檔案證據與行號，不得直接歸因於使用者。

5. 不得把 assistant 自己的版本對齊、解壓、暫存、容器、套件、資料遺失或測試操作問題歸因於使用者。

## D. 資料集與執行環境規則

1. 動態測試預設只要求 `tw_stock_data_vip_reduced`，不要求 `tw_stock_data_vip`。
2. 若執行環境存在 `/data`：
   - `apps/validate_consistency.py` 預設使用 `/data/tw_stock_data_vip_reduced`
   - `apps/portfolio_sim.py`、`apps/vip_scanner.py`、`apps/ml_optimizer.py` 預設使用 `/data/tw_stock_data_vip`
3. 若執行環境不存在 `/data`，則退回專案根目錄：
   - `data/tw_stock_data_vip_reduced`
   - `data/tw_stock_data_vip`
4. 不得把外部環境剛好存在的 `/data/...` 視為專案內建資產。
5. 若需外部套件（如 `optuna`），應依 `requirements` 自行安裝後再驗證；不得因缺套件就略過本應執行的動態測試。
6. 除 consistency 外，一般 UI 不需額外詢問資料集；若需切換，只能透過 CLI `--dataset` 或環境變數覆蓋。

## E. Coding 原則

1. 單一真理來源：相同邏輯不得在不同地方重複實作。
2. 統計口徑必須完全一致：回測、優化、模擬、驗證工具中的成交、未成交、miss buy、EV、勝率、Round-Trip PnL 定義不得分叉。
3. 避免 magic number；公式與參數必須具數學意義與可解釋性。
4. 禁止裸 except；所有異常必須可追蹤。
5. 架構調整不得明顯犧牲效率；若提高未來策略修改或 ML / DRL / LLM 升級複雜度，必須先明確說明。
6. 正式執行入口集中於 `apps/`；`core/` 只放核心規則與共用計算；`tools/` 只放驗證、除錯與開發輔助工具。
7. 拆分、合併、移動、重新命名檔案時，必須遵守：
   - 單一職責
   - 上層呼叫下層
   - 禁止反向依賴
   - 禁止循環依賴
   - 禁止規則分叉
   - 禁止重複實作
8. 新增模組與檔名應優先使用職責語意命名，不得再新增版本前綴式命名。
9. 若架構或模組責任有變動，必須同步更新：
   - `doc/ARCHITECTURE.md`
   - `doc/CMD.md`
10. 若架構改動後舊入口、舊 wrapper、舊 helper、舊檔名或舊路徑已不再屬於正式結構，必須明確列出應刪除的舊檔與舊引用；不得長期新舊並存。

## F. 交易與統計原則

1. 杜絕未來函數：不可偷看未來資料。
2. 同 K 棒同時觸發停損 / 停利時，一律以最壞停損情況計算。
3. 資金與權益曲線一律以扣除手續費、稅金後的淨值為準。
4. 半倉停利僅視為現金回收，尾倉結算才算完整 Round-Trip PnL。
5. 只能盤前掛單；盤中不得新增、改單、換股。
6. 買入股數上限、總資金占用、最大持倉數、單筆風險曝險，都必須在盤前即可確定。
7. 不得當沖；買入當日不可賣出。
8. 當日賣出回收資金，不得於當日再投入。
9. 若某檔當日未成交，不得於同一交易日盤中自動改掛其他股票。
10. 停利 / 停損只能對已持有部位預先設定。

## G. 候選、延續候選、miss buy 定義

1. 當日買入候選：同時符合技術買入訊號與歷史績效門檻。
2. 每日候選池由新候選與延續候選共同組成。
3. 是否掛單由盤前排序、可用資金、持倉名額、風險股數上限共同決定。
4. 延續候選：先前出現技術訊號但尚未買入者，後續只要仍符合當日延續條件、歷史績效門檻與風險限制，即可重新參與排序。
5. 前一日未被選上、資金不足、名額不足、或有效掛單未成交者，後續只要仍符合延續條件，皆可重新參與排序。
6. 延續候選不是保送通道；不得因曾入池、曾掛單、曾未成交，就繞過當日排序直接買進。
7. miss buy：已於盤前保留資金與名額，且已形成有效限價買單，但盤中未成交。
8. 以下都不算 miss buy：
   - 排序未選上
   - 資金不足
   - 持倉名額不足
   - 風險股數為 0
   - 先達停損而放棄進場
9. 候選資格、是否掛單、是否成交、是否 miss buy，必須分開定義，不得混用。

## H. 策略邏輯

1. 單股回測不得使用該檔自身歷史績效 filter 作為買入閘門。
2. 單股回測必須完整依技術訊號、盤前掛單、成交 / 未成交、停損 / 停利規則產生交易歷程。
3. 歷史績效 filter 只能用於投組層的候選資格、排序、是否掛單；不得回頭影響單股回測的交易生成。
4. 投組層的歷史績效統計來源，必須來自單股回測產生的完整交易結果；不得以投組自身是否實際買到作為統計母體。
5. 技術訊號、候選資格、是否掛單、是否成交、miss buy、歷史績效統計，必須分層定義，不得混用。
